## What's in each JSON

Every preset's JSON contains:

```json
{
  "timestamp_utc": "...",
  "preset_id": "01_sanity",
  "machine": {"platform": "...", "machine": "...", "python": "..."},
  "config": {
    "dataset": "...", "n_values": [...], "d_bits": ..., "k_values": [...],
    "methods": [...], "estimated_memory_mb": ..., "total_elapsed_s": ...
  },
  "results": [
    {"method": "...", "time_s": ..., "pairs": ..., "recall_vs_truth": ..., "extra": {...}},
    ...
  ]
}
```

These are the reproducible primary source. Anyone with the demo can re-run
the same preset on different hardware and produce a comparable JSON.

## To reproduce on different hardware

1. `pip install streamlit faiss-cpu usearch annoy hammingbird`
2. `cd hammingbird && streamlit run code/demo/app.py`
3. Pick each preset → Run. Each Run writes a new JSON to `logs/demo/`.
4. `python3 code/demo/bench_threads.py` for the rayon scaling characterization.

JSONs can then be diff'd or aggregated against the ones here.

## Headline numbers (see `../../BENCHMARK_RESULTS.md`)

The full head-to-head writeup lives at `hammingbird/BENCHMARK_RESULTS.md`
(repo root). The most pitch-relevant single numbers:

- **4073× over FAISS Flat** at n=2M, k=0 (exact dedup, the most common
  production workload)
- **0.21 µs median Index query latency** at n=100k (streaming primitive)
- **45× over FAISS MultiHash at full recall** on clustered data
  (the same algorithm, our implementation)
- **Strict dominance over usearch/Annoy/FAISS-LSH** on clustered data
  at all sizes tested (n=100k–1M): pigeon is faster AND more accurate.
